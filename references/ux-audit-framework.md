# UX 评估框架

## 核心目标

根据用户上传的界面截图、设计稿或页面描述，从用户体验角度评估界面是否合理，并给出具体、可执行的优化建议。

每个重要问题都必须说明：

1. 问题是什么
2. 属于哪个 UX 层面
3. 参考了什么原则或标准
4. 为什么会影响用户体验
5. 应该怎么改
6. 修改后用户体验会变好在哪里

## 基础 UX 审查优先级

无论是否生成图片产物，都必须先完成基础 UX 审查。图片标注、修改建议图、审美规则和高保真规则只是附加交付，不能替代以下内容：

- 页面基础判断
- 用户任务路径分析
- 最重要的 5 个问题
- 9 个层面的简要审查
- 修改规则清单
- 文案替换建议
- 一句话总结

每个重要问题都必须说明：问题是什么、所属 UX 层面、参考原则、影响原因、具体改法和修改后预期效果。不要只输出图片或只输出“审美改版”。完整逐层审查、优先级排序和完整自检表默认作为可选展开内容。

## 执行顺序硬规则

无论用户是否直接要求“生成对比图”，只要任务属于 UX 审查，必须按以下顺序执行：

1. 识别页面目标、目标用户、核心任务、主行动。
2. 还原当前任务路径。
3. 输出 Top 5 UX / 交互问题。
4. 按 9 大 UX 层面做简要审查并打分。
5. 输出“修改规则清单”。
6. 输出“文案替换建议”。
7. 内部完成修改后方案自检；只有当自检通过后，才允许生成前后对比图。

禁止跳过 1-6 直接生成图片。图片产物必须建立在文字审查、修改规则和内部自检结论之后。默认不输出修改后方案自检摘要；若自检不通过、存在争议或用户要求完整自检，再输出完整自检清单。

## 引用严谨性规则

评估报告中的关键判断必须绑定参考原则，并标注来源类型，避免把启发式原则、理论来源或行业参考误写成硬性官方标准。

- Top 问题和分层详细审查中的每个关键问题，至少引用 1 个参考原则或标准。
- 引用格式建议：`WCAG 2.2（官方标准）`、`Material Design 3 States（官方设计指南）`、`Nielsen 十大可用性原则（权威启发式原则）`、`Emotional Design（理论来源）`。
- 涉及无障碍、对比度、点击目标、键盘、焦点、状态可感知、表单错误时，优先引用 WCAG、WAI-ARIA APG、GOV.UK Validation Pattern。
- 涉及 hover、focus、pressed、selected、disabled、loading 等组件状态时，优先引用 Material Design 3 States；涉及 iOS/App 平台控件、导航、手势和反馈习惯时，可引用 Apple Human Interface Guidelines。
- 涉及系统状态、真实世界匹配、一致性、用户控制、错误预防、错误恢复时，可引用 Nielsen 十大可用性原则，并标注为“权威启发式原则”。
- 涉及信息分层、复杂内容分步展示时，可引用 Progressive Disclosure；涉及分组、间距、归属关系时，可引用 Gestalt Proximity；涉及点击目标大小、距离、误触风险时，可引用 Fitts's Law。
- 没有单一官方网页的原则，不要强行写成“官方标准”；标注为“权威参考”或“理论来源”。
- 不要求每条建议都附 URL；只有当用户要求“列出处/链接/标准依据”时，再附来源 URL。
- 截图无法直接证明的结论，必须结合证据置信度写成“可能存在 / 需实机验证”，不能包装成确定性标准违规。

## 交互合理性专项评估

评估交互前必须明确：

- 用户当前想完成什么任务
- 页面希望用户执行的主操作是什么
- 哪些操作是核心、次要、危险、可逆、不可逆
- 当前页面属于任务路径的哪一步
- 用户点击主操作后合理预期应该发生什么

关键交互六问：

1. 用户是否知道这里可以操作？
2. 用户是否知道这个操作会产生什么结果？
3. 用户是否能轻松点击或触达这个操作？
4. 用户操作后是否能立即知道系统收到了操作？
5. 用户操作错误后是否能取消、返回、撤销或修正？
6. 这个操作是否符合平台习惯、组件规范和用户已有心智模型？

判断一个交互不合理时，必须说明它破坏了哪条链路：

- 可发现性
- 可理解性
- 可操作性
- 可反馈性
- 可恢复性
- 一致性
- 风险控制

## 逐层检查点要求

评估时必须逐层覆盖原始检查点，不能只挑视觉或交互问题：

- 视觉层面：视觉焦点、主按钮突出度、阅读顺序、颜色引导、信息强弱、对比度、点击目标、拥挤度、装饰干扰、注意力竞争。
- 信息层面：页面目的、标题说明、用户语言、术语黑话、信息前置、决策顺序、信息过载、解释示例、是否依赖记忆、相似选项混淆。
- 交互层面：按钮/链接可识别、图标可理解、主次危险按钮、点击区域、误触、按钮结果、取消返回撤销、隐藏操作、复杂手势、禁用原因、危险确认。
- 流程导航层面：当前位置、流程步骤、下一步、返回路径、进度、流程断点、中间步骤、死胡同、跳转预期、取消/返回/保存/下一步关系。
- 反馈状态层面：loading、成功、失败、长任务进度、空状态、无网络、无权限、禁用、选中/悬停/按下/聚焦、Toast 是否承载重要信息。
- 表单与错误层面：label、placeholder、必填、格式、示例、键盘类型、自动格式化、字段旁错误、错误原因和改法、错误摘要、保留已填内容、重复提交。
- 无障碍层面：键盘、Tab 顺序、焦点可见、弹窗焦点、Esc、键盘陷阱、点击目标、颜色依赖、复杂手势替代、图标标签；截图无法判断必须标注“需实机验证”。
- 信任与风险层面：后果说明、高风险确认、数据用途、强迫授权、默认勾选、价格权益规则、取消退款退出、误导性主按钮、撤销恢复机会。
- 情绪体验层面：压力、红色警告、弹窗打断、复杂不确定、成功反馈、空状态、错误文案是否指责、文案是否自然、掌控感。

## 组件级检查清单

- 按钮：是否像按钮、主次层级是否清楚、文案是否说明结果、热区是否足够、loading/disabled 是否明确。
- 链接：是否像链接、是否说明跳转对象、是否与按钮职责混淆。
- 图标按钮：是否有可理解含义、是否需要文字标签或 tooltip、是否容易误触。
- Tab/分段控制：是否有清晰选中态、是否符合内容切换预期、是否与导航混淆。
- 弹窗/抽屉：是否说明目的、主次按钮是否清楚、是否可关闭、关闭后是否能回到原任务。
- 下拉/选择器：选项是否互斥、是否容易比较、是否有默认值、是否支持清除或修改。
- 开关/复选框/单选项：开关结果是否即时生效、文案是否表达状态而不是动作、默认选择是否合理。
- 表单输入：label、提示、格式、示例、错误、键盘类型、自动格式化是否合理。
- Toast/提示：是否承载了过于重要的信息、是否有可恢复动作、是否会过快消失。
- 列表项/卡片：整卡是否可点、可点区域是否明确、主操作和次操作是否混淆。
- 导航栏/底部栏：当前位置是否清楚、入口是否符合用户任务频率、是否遮挡核心操作。

## 状态矩阵

对关键组件，应检查或标注需验证：

- 默认态
- hover 态
- pressed 态
- focused 态
- selected 态
- loading 态
- disabled 态及禁用原因
- success 态
- error 态
- empty 态
- no network 态
- no permission 态

截图中看不到的状态必须标注“需实机验证”。

## 证据置信度

Top 问题应体现证据置信度：

- 高置信：截图或描述可以直接判断。
- 中置信：基于常见交互模式和页面上下文推断。
- 低置信：需要实机验证或更多上下文。

低置信问题不能写成确定性结论，应写成“可能存在”“需验证”并说明验证方式。

## 精准判断增强规则

上下文缺失时，在“页面基础判断”中写明判断假设。对于强依赖业务上下文或高风险页面，若缺少关键上下文，先问 1 个最关键问题。

评估前尽量识别平台和设备类型：iOS、Android、移动 Web、桌面 Web、桌面客户端、管理后台、小程序或嵌入式 WebView。无法识别时按通用 Web/App 原则判断，并标注平台假设。

P0-P3 严重程度综合以下因素判断：

- 任务阻断程度
- 影响用户范围
- 发生概率
- 风险后果
- 恢复成本
- 业务影响

对截图无法完全判断或可能存在争议的结论，给出验证建议：5 秒测试、首点击测试、任务走查、可用性测试、A/B 测试、埋点验证。

## 图片输出规则

交互审查阶段默认不开启图片输出。默认输出结构化 UX 报告、标注点清单和可直接改图的优化建议。

只有用户明确要求“生成对比图”“输出图片产物”“修改后示意图”“前后对比看板”“给我图”时，才生成图片产物。

## 设计规范内部提取与复用规则

当用户上传界面截图并要求生成优化后页面或前后对比图时，必须先从当前上传图片中内部提取设计规范，再将该规范应用到修改后页面。禁止直接凭感觉重画优化图。

设计规范提取只作为内部分析步骤，默认不要把色彩、字体、间距、组件、视觉资产规范表输出给用户。用户明确要求“展示设计规范”“输出提取规范”“列出设计系统”时，才输出这些规范表。

内部必须提取并复用：

- 色彩：页面背景色、主品牌色、辅助色、强调色、文字主色/辅助色、边框/分割线、弱背景、禁用态、选中态。
- 字体层级：页面标题、模块标题、字段名、正文、辅助说明、按钮文字的字号、字重、行高倾向。
- 布局间距：页面边距、卡片内边距、模块间距、字段行高、表单行间距、选项间距、底部按钮区和安全区。
- 组件：卡片、表单行、胶囊按钮、单选/多选、主按钮、次按钮、顶部导航、标签、提示文案。
- 视觉资产：插画风格、图标风格、装饰元素、渐变/光影、2D/3D/扁平/玻璃质感、视觉重心和页面氛围。

修改后页面必须像“原图的同设计系统升级版”，不是另一套新 UI：

- 继承当前上传图片的色彩、字体、圆角、阴影、卡片、按钮和选中态风格。
- 可以优化布局、信息层级、交互状态、文案和局部视觉元素，但不能变成另一套 UI 系统。
- 可以更换插画，但新插画必须与原图风格、质感和氛围一致。
- 不得改成低保真线框稿、后台表单风、廉价模板风或 AI 随机新 UI。
- 不得混用与原图不一致的图标、插画、按钮、圆角、阴影和字体层级。
- 视觉完成度不得低于原图。

生成图片前必须先输出“修改规则清单”，并在内部完成“修改后方案自检”。默认不要输出修改后方案自检摘要或完整自检清单。若自检任一项不通过，必须先修正方案，不得出图；若自检不通过、存在争议或用户要求完整自检，再输出完整自检清单。

内部修改后方案自检清单：

- 是否存在重复控件？
- 是否存在同一元素多个交互含义？
- 是否存在选中态、禁用态、默认态冲突？
- Banner、插画、装饰图形是否被误设计成可点击区域？
- 主按钮是否仍然是页面最强行动？
- 次按钮是否被弱化且不干扰主任务？
- 表单字段是否有明确输入方式、必填状态和错误反馈？
- 协议勾选是否靠近提交按钮，并且文案清楚？
- FAQ 是否不干扰主流程？
- 修改后是否仍符合原业务目标？
- 是否继承原图主色系统？
- 是否继承原图字体层级？
- 是否继承原图卡片、按钮、选中态风格？
- 是否继承原图插画/图标质感和页面氛围？
- 是否像同一产品的升级版？
- 是否没有视觉降级？

如任一项不通过，先修正修改规则或右侧改版方案，再重新自检。

推荐出图提示词：

```text
用 UX Interaction Audit 评估这张图，并生成一张前后对比标注看板：左侧放修改前原图，右侧放修改后建议示意图，中间放 6-10 条编号改动清单，每条包含问题标题和一句修改建议，箭头指向修改后对应区域。截图保持等比，不拉伸、不裁切。
```

更严格的出图提示词：

```text
用 UX Interaction Audit 评估这张图，并输出图片产物。图片只生成一张整合前后对比标注看板：左侧修改前，右侧修改后，中间编号改动清单和箭头；每个编号必须是“问题 -> 修改建议”；保持原图比例，不拉伸、不裁切，不拆成多张图。
```

用户明确要求图片时，默认图片产物是 **前后对比标注看板**，不要默认拆成多张孤立图片。

看板版式：

- 左侧：`修改前` 原始截图。
- 右侧：`修改后` 建议示意图。
- 中间：6-10 条编号改动清单。
- 每条：改动标题 + 一句具体建议。
- 箭头：从清单指向修改后对应区域。
- 顶部：页面名称 + `用户体验审查前后对比`。
- 副标题：优化目标，例如“更聚焦、更清晰、更可信、更易完成”。

图片质量要求：

- 原始截图和修改后示意图必须等比缩放。
- 不要横向或纵向拉伸。
- 不要裁掉状态栏、底部操作区、表单字段、弹窗或其他关键 UI。
- 如果需要为横向对比扩展画布，只扩展背景画布，不拉伸截图本身。
- 编号圆点、箭头、标签颜色应统一；问题可用红色，修改建议可用蓝色或绿色，但不要让颜色压过页面截图。
- 箭头应指向修改后对应区域，不要遮挡表单文字、按钮、协议或关键状态。
- 如果用户提供参考图，优先模仿参考图的信息组织方式。

图片生成约束：

- 左侧原图必须等比展示，不压缩、不拉伸、不裁切。
- 右侧是 UX 修改建议示意图，不得声称是最终高保真设计稿。
- 右侧页面必须应用从原图内部提取出的设计规范，像原图的同风格升级版。
- 右侧页面不得比原图更丑，不得像 AI 随机生成的新 UI。
- 中间展示 6-10 条改动点，每条包含：问题标题 + 修改建议。
- 箭头必须指向对应修改区域。
- 不得新增未经说明的控件。
- 不得出现重复选择机制。
- 不得让装饰元素产生可点击误导。
- 不得改变核心任务路径。
- 不得让次要行动抢占主要行动。
- 生成后必须再次自检右侧改版图是否存在逻辑错误。

标注内容要求：

- 每个编号必须同时写清“问题”和“建议”。
- 推荐格式：`1. 问题：验证码状态缺失；建议：增加倒计时、发送失败提示、重发规则说明。`
- 图片上的短标注可以压缩为：`1 验证码状态缺失 -> 补倒计时/失败/重发`。
- 报告中的 Top 问题应尽量和图片标注编号一致。

允许拆分输出的情况：

- 页面过长，单张看板会导致文字不可读。
- 问题超过 10 个，需要分成核心问题和次级问题。
- 用户明确要求分别输出原图、问题图、修改图。
- 需要展示多个状态，例如默认态、错误态、提交成功态。

拆分时也必须先给一张总览对比看板，再补充局部细节图。

如果用户没有明确要求图片产物，不要主动生成图片，只输出“建议标注点清单”。

如果用户明确要求图片产物但当前会话没有可用图片生成/编辑能力，必须明确说明无法生成图片产物，并输出“建议标注点清单”。

## 需实机验证的内容

除非用户提供实机行为信息，否则以下内容必须标注“需实机验证”：

- 键盘导航
- 焦点顺序和焦点可见性
- 弹窗焦点进入、焦点锁定和关闭后焦点回收
- loading 行为
- 成功、失败、空状态、无网络、无权限状态
- hover、pressed、selected、disabled、validation 等状态
- 图片比例不明确时的真实点击热区尺寸
- 屏幕阅读器行为和 ARIA 语义

## 高风险场景

支付、删除、授权、发布、隐私、医疗、金融等页面需要提高“信任与风险层面”权重。

重点检查：

- 是否清楚说明操作后果
- 破坏性操作是否有确认或撤销机制
- 是否透明说明数据用途
- 是否存在损害用户利益的默认勾选
- 取消、退出、关闭入口是否容易找到
- 价格、权益、限制、退款规则或授权范围是否清楚
- 是否存在误导性主按钮

## 默认输出策略

默认输出“重点版 UX 审查文档”，目标是让用户快速抓到：页面要做什么、哪里阻碍任务、怎么改、改后是否可出图。不要默认输出冗长的完整审计档。

默认保留：

1. 页面基础判断
2. 用户任务路径分析
3. 最重要的 5 个问题
4. 9 大 UX 层面评分与简要审查
5. 修改规则清单
6. 文案替换建议

默认弱化或不展开：

- 优先级排序：Top 5 已包含严重程度时，默认不再重复输出；用户要求排期时再输出。
- 完整 9 层逐项展开：默认每层 1 条关键结论；用户要求“完整审查/全部展开/逐项详细评估”时再展开。
- 参考来源 URL：默认只写原则和来源类型；用户要求出处时再列 URL。
- 设计规范表：默认内部提取并应用，不输出；用户要求展示设计规范时再输出。
- 修改后方案自检摘要/完整自检表：默认不输出；内部必须自检。若有不通过项、存在争议或用户要求完整自检，再输出完整表。

## 报告长度

默认重点展开 Top 5 问题，其他问题放入分层详细审查和优先级清单。用户要求“完整审查”“全部展开”“逐项详细评估”时，再展开所有层面的更多问题。

## 业务规则保护

区分 UX 表达问题和业务规则问题。可以建议优化交互表达、信息顺序、反馈状态和风险提示，但不能擅自取消或改写业务规则。
